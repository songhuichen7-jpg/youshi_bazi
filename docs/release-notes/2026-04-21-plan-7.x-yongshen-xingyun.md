## 一句话总结

把 `suggest_yongshen` 这个 4 行启发式函数，换成了两个经典古籍驱动的真引擎：**Plan 7.3** 按三法（调候/格局/扶抑）定命局用神并带出处，**Plan 7.4** 给大运流年对照用神打 5-bin 评分。LLM 现在回答 "我用神是什么" "我 2030 年怎么样" 这类问题时，不再靠通用命理知识幻觉，而是看着结构化的引擎输出引经据典。

## 用户感知变化

### 以前（Plan 7.2 及之前）

用户问 "我的用神是什么？"：
```
你的用神可能是甲木。根据丁火生于六月的特点...
```
LLM 全程基于自己对古籍的通用知识回答，没有可追溯的引证，经常在细节上漂移。

用户问 "我 2030 年怎么样？"：
```
2030 年是庚戌年，对丁火日主来说，庚金是...
```
LLM 不知道用户的命局用神是什么，不知道这步大运对命局的影响，全靠年柱干支单独判断。

### 现在（Plan 7.3 + 7.4 上线后）

用户问 "我的用神是什么？"：
```
先说最要紧的调候。你是丁火日主，生在未月，农历六月。《穷通宝鉴》里
讲得非常直接：「六月丁火，阴柔退气，专取甲木，壬水次之。」白话就是，
六月的丁火像夏末快要熄灭的烛光，气势已经弱了，最需要的就是甲木来
当柴火...

再看格局用神。你的月令未土，本气是己土食神，天干也透出己土，所以
是食神格。《子平真诠》论食神生财时说：「财喜食神以相生」...
```
LLM 看着引擎注入的三法用神 + 具体古籍章节页码回答，引证精确到月份。

用户问 "我 2030 年怎么样？"：
```
2030年庚戌，是你这步乙卯大运里最需要留神的一年。

天干乙庚合，这是今年最大的事... 酉戌相害：戌来害你年支和日支的酉
金偏财。财星被害，最直接的体现就是合同纠纷、投资踩坑... 2032年
壬子年的好运就在前面不远处...
```
LLM 看着引擎给出的 `⚠忌 庚克用神` + `⭐⭐大喜 壬生用神，与命局合化木转助` 等评分，写出有具体年份锚点和转折判断的叙事。

### 关键差别

> 引擎是 floor，LLM 是 ceiling。

引擎提供结构化、可追溯、古籍引证的定量判断；LLM 把这些判断包装成命理师口吻的叙事。用户拿到的回复既有**专业命理师的严谨**，又有**资深咨询师的温度**，同时每句关键判断都可以往回追到具体经典章节。

## Plan 7.3 — 用神引擎 v1

### 核心

命局用神的三法合成：
- **调候法**（`穷通宝鉴·论十干·论X月`）— 120 条数据，10 日干 × 12 月令全覆盖
- **格局法**（`子平真诠·论X格`）— 12 个格局 × 平均 3 条规则 = 约 30 条决策规则
- **扶抑法**（`滴天髓·衰旺`）— 5 个 dayStrength 分支（极弱/身弱/中和/身强/极强）

三法独立计算，`compose_yongshen` 按 voting rule 合成：
- 调候 == 格局 → primary = 调候.name，无 warning
- 调候 != 格局 → primary = 调候.name + warning "古籍两派各有取法"
- 仅格局 → primary = 格局.name
- 仅扶抑 → primary = 扶抑.name
- 三法皆空 → primary = "中和（无明显偏枯）"

### 接口

```python
chart.paipan.yongshen         # str — 主用神字符串（chartUi.js 向后兼容）
chart.paipan.yongshenDetail   # dict — 完整三法候选 + 出处 + warning
  ├─ primary: str
  ├─ primaryReason: str
  ├─ candidates: [{method, name, note, source}, ... 3 条]
  └─ warnings: [str, ...]
```

### 经典来源（实打实读过）

| 古籍 | 章节范围 | 用途 |
|---|---|---|
| 穷通宝鉴 | 10 个日干分论（02-11）| 调候 120 条 |
| 子平真诠 | 8, 9, 16, 31-47 | 格局 30 条规则 |
| 滴天髓 | 衰旺、中和、体用、顺逆 | 扶抑 5 分支 |

### 关键 catch（审核过程中发现）

- **`_GEJU_ALIASES` 别名层**（Task 3）— paipan 的 ge_ju.py 对比肩月令输出 `建禄格`、劫财月令输出 `月刃格`/`阳刃格`，但 GEJU_RULES 的 key 是 `比肩格`/`劫财格`。如果没有 alias 层，**所有比肩/劫财月令盘的格局法会静默失效**。这是 review 过程从 ge_ju.py 源码反推出来的。
- **癸辰 TIAOHOU 条目 cosmetic fix** — 初版 name 写成 `'丙/丙'` 冗余重复；壬子条目的 name 顺序跟 source note 里的"先戊后丙"矛盾。修正后与经典一致。

## Plan 7.4 — 行运引擎

### 核心

对照 Plan 7.3 确定的命局用神，给每条大运（8 条）和每个流年（8 × 10 = 80 条）打一个 5-bin 喜忌评分 + 1 行 note + 机制标签。

**范围**（刻意小）：命局用神**不变**。7.4 只算"行运干支对命局用神的作用"。用神本身随大运三合三会改月令而质变（ZPZQ ch10 「论用神变化」）留给 Plan 7.5。

**评分机制（层 2）：**
1. **干 effect**（基础 ±2）：行运干五行与用神五行的比助/相生/相泄/相克关系
2. **支 effect**（基础 ±2）：行运支本气五行同上
3. **干合化 modifier**（±1）：行运干与命局透干构成的五合（甲己→土等）
4. **六合 modifier**（±1）：行运支与命局支构成的六合（子丑→土等）
5. **5-bin 分类**：`>=4 大喜 / 2-3 喜 / -1..1 平 / -3..-2 忌 / <=-4 大忌`

**多元素用神**：primary 如果是 `甲木 / 戊土 / 庚金` 多用神同辉，对每个元素算 sub_score，**取最大**作为最终分（"任一用神被生扶即主事"的古籍直觉）。

**中和命局特殊处理**：primary 是"中和（无明显偏枯）"时，engine 仍跑但全部返回 `平` + 空 mechanisms，render 层整段折叠不渲染（避免 LLM context 充斥无效信息）。

### 接口

```python
chart.paipan.xingyun
  ├─ yongshenSnapshot: str               # 评分对照的用神（== yongshenDetail.primary）
  ├─ currentDayunIndex: int | None       # 当前大运 index（1..8）
  ├─ dayun: [                             # 8 条
  │     {index, ganzhi, startAge, startYear, endYear,
  │      label, score, note, mechanisms, isCurrent},
  │     ...
  │  ]
  └─ liunian: {                           # 键 "1".."8"
        "1": [{year, ganzhi, age, label, score, note, mechanisms}, ... 10 条],
        ...
     }
```

### 关键 catch

- **1..8 vs 0..7 index**（Task 6）— 我的 plan 假设 dayun index 是 0..7，但 lunar-python 的 `getIndex()` 返回 1..8（因为 `raw_dayun[1:9]` 跳过了起运前那一条）。Codex 对照真实数据改正。
- **`_classify_score` 边界 off-by-one**（Task 2）— plan 里 cascade 用了 `SCORE_THRESHOLDS['忌']=-2` 作为 忌 分支的下限，但 spec §3.4 规定 -3..-2 都是 忌。这会让 score=-3 静默滑到 大忌。Codex 改用字面 `-3` 硬编码修正。
- **sizhu vs bazi key + 未知时辰防御**（Task 1）— plan 里的 `result["bazi"]` 实际 key 是 `result["sizhu"]`；同时 sizhu.hour 未知时辰时是 `None`，需要 `if bazi.get(k)` 防御 guard，否则未知时辰盘会 TypeError。

这 3 个 catch 都是 codex 读真实代码自己发现的，plan artifacts 按每次 catch inline 修正过。

### 经典来源

- 五行生克基本表 — 历代古籍通论
- 天干五合（甲己→土等） — 通论
- 地支六合（子丑→土等） — 通论
- 5-bin 映射命名 — 用户直觉

> 注：7.4 的评分机制不像 7.3 可以精确引到某一章节页码（行运喜忌在古籍里没有单章专论，散见多处）。mechanism 标签用 `干·相生`/`支·六合化木·转助` 这种结构化短语替代。

## 工程方法学

### Spec → Plan → 外部执行 → Review 的流程

两个 plan 都走了同一个流水线：

1. **Brainstorming skill** — 一次一个问题，B 选项优先，用户决策 5-6 次收敛成完整 design
2. **Spec 文档**（`docs/superpowers/specs/YYYY-MM-DD-X-design.md`）— 12 节 + 2 个 Appendix
3. **Writing-plans skill** — 7 task，每 task 2-8 个 bite-sized step，完整代码 + 确切命令 + commit message 模板
4. **Codex 外部执行** — 每次一个 task，自己跑 tests + 自己 commit，report SHA + sanity check 输出
5. **Review（我）** — 对 report 做 spec conformance + sanity math 核对，发现 plan bug inline 修正，批准 push
6. **用户 push** — `git push origin main`

整个流程让**实施和审核完全解耦**：codex 不知道前一轮的决策历史，每次只读当前 plan 和 spec；我不写具体代码，只读 codex 的 commit + report。这样 codex 不会从 conversation history 偷懒，我不会被实施细节淹没。

### 一个月内的节奏

这次 2 个 plan 连着做：
- 4-20 上午 Plan 7.3 spec + plan
- 4-20 白天 Plan 7.3 执行（7 tasks，codex 一个接一个）
- 4-20 晚上 Plan 7.4 spec + plan
- 4-21 Plan 7.4 执行（7 tasks，同节奏）

两个引擎合计 ~20 个 commits，73 个新测试，~800 条结构化数据条目，约 2000 行新代码。

### 关键的 review habits

1. **Sanity math 前置预测** — 在给 codex 的 prompt 里预先手算一条数据（例如 "对于 yongshen=木 + 行运庚申，应该 delta=-3 + 合化金反克"），codex 跑完把真实输出贴回来字符对比。这种 pre-registered prediction 让 "他 implementation 跑通了但逻辑错了" 这种假阳性几乎无处藏。
2. **Plan artifact 每次 catch 都改** — codex 发现 plan 里写的 `d.gan` 应该是 `d["gan"]`，我不是嘴上"好的好的"，是 inline 改 plan，然后 `docs(plan): fix ...` 单独 commit。后续 task 读到的 plan 是修正过的。
3. **Disclosure 强制** — prompt 末尾永远写 "If X happens, disclose don't silently fix"。codex 每轮都老老实实说明自己偏离 plan 的地方（Task 4 "compute.py doesn't expose result['bazi']，我用 sizhu"、Task 6 "dayun index 是 1..8 不是 0..7" 等）。
4. **Golden case 是 structural，不是 oracle** — 10 个 birth_input 只断言 "有 8 条大运、label ∈ valid set、mechanisms 非空"，不断言具体数值。否则 codex 会按测试数值反推实现，把 bug 硬塞进测试里。

## 限制 + Reserved 后续

### Plan 7.3 已知限制

1. **TIAOHOU 阈值/边枝精修** — audit 时有 2 个条目 `丁巳`/`丁寅` 是 judgment call，primary/supporting 顺序有争议。暂时按 codex 的读法 ship，Plan 7.4.1 可以重读经典细化。
2. **`_names_match` 严格字符串比较** — "庚金" 和 "庚" 不算共指。以后可以加五行等价比较，减少 warning 误报。
3. **GEJU_RULES 5 类边枝没加** — 七杀财先食后、印多用财、杂气月透与会支、财格佩印、七杀化印借财。都需要扩 analyzer 的 `force` + `gan_he` 输入维度。

### Plan 7.4 已知限制

1. **合化任意位置简化** — 相邻才严格算合化，当前不判相邻。准确度损失约 5-10%。
2. **多元素用神取 max 偏乐观** — 可以改 weighted average。
3. **极弱/极强 dayStrength forward-compat** — li_liang.py 当前只 emit 身弱/中和/身强 3 个值；FUYI_CASES 里极弱/极强 2 个分支暂时是 dead code，等 Plan 7.4.1 升级 li_liang 激活。
4. **大运-流年 cross interaction 评分** — 当前只算行运干支 vs 命局干支。大运 乙 × 流年 庚 合化金这种 cross 作用没进 engine（LLM 靠自己的经典知识补）。
5. **流月评分** — 粒度止于流年。
6. **反激/通关/调候 modifier** — 层 4 未实施，留给 Plan 7.5。

### Plan 7.5（Reserved）

**用神变化** — ZPZQ ch10 strict reading：大运/流年带三合三会或透藏改月令时，**命局用神本身**换候选。例：丁火生亥月本为正官格，大运卯 + 命局含未 → 亥卯未三合木局 → 月令性质变成印，用神跟着换。跟 7.4 的"评分不变用神"形成完整的两层时间轴用神系统。
