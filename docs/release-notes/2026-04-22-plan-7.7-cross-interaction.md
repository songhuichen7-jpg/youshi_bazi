## 用户感知变化

### 以前（Plan 7.6 及之前）

用户问 "**这步大运里某些年份，是不是有干支跟我大运干合化？**"，LLM 只能凭通用命理知识回答：

> "理论上 2026 丙午年的丙跟你大运辛可能形成丙辛合，但需要进一步分析..."

—— LLM 看 chart context 里的 mechanism 信号，发现没有 "干·合化转助·水" 之类的 cross fire（因为 Plan 7.4 score_yun 只看 流年 vs 命局），所以只能凭经验 hedge。

### 现在（Plan 7.7 上线后）

引擎 mechanism 层直接 fire `干·合化转助·水`（流年丙 + 大运辛 = 丙辛合化水）。LLM 看到这个 tag, 自然写：

> "这年流年天干丙，与你大运天干辛，形成了丙辛合。这个合，按五行理论是想化水的。但能不能化成，要看条件...用神壬水在原局没透干，藏在戌土里，所以这个合化水很难成功，更多是'合绊'的状态。"

—— 直接确认 cross 合化存在 + 引古籍做 second-order 推理（化不化得成）。Engine 给 raw signal，LLM 做 nuance。

### 补充亮点

同一段 chat reply 里 LLM 还自然 cite 第二个 cross interaction：

> "2028 戊申年... 这年流年地支申，与你大运地支巳，形成了巳申合。"

`巳申合` 也是 cross modifier (流年支 vs 大运支)。Plan 7.7 一次扩展同时激活了**干** + **支** 两层 cross。

## 核心机制

### 单点修改

`paipan/paipan/xingyun.py` build_xingyun() 流年 loop:

```python
# Plan 7.4 旧:
for ly in entry.get('liunian', []):
    ly_score = score_yun(ly['ganzhi'], yongshen_primary, mingju_gans, mingju_zhis)
    ...

# Plan 7.7 新:
for ly in entry.get('liunian', []):
    extended_gans = mingju_gans + [ganzhi[0]]   # +大运干
    extended_zhis = mingju_zhis + [ganzhi[1]]   # +大运支
    ly_score = score_yun(ly['ganzhi'], yongshen_primary, extended_gans, extended_zhis)
    ...
```

3 行 diff（1 行调用替换 + 2 行赋值）。score_yun 接口不变。

### 范围分离

Plan 7.7 cross interaction 只改**流年 score 评分层**。不改：
- **大运 score** (大运 vs 命局, Plan 7.4 行为不变)
- **Plan 7.5b transmutation 检测** (仍用 base mingju_zhis, 大运/流年 transmutation dedup logic 自己处理大运参与)

两层 separate, 互不污染。

### Mechanism tag 复用，无新 namespace

Plan 7.4 + 7.6 的 14 个 mechanism tag (`mechanism_tags.py`) 全复用：
- `gan_hehua_zhuanzhu(wx)` → `'干·合化转助·{wx}'`
- `zhi_liuhe_fanke(wx)` → `'支·六合化{wx}·反克'`  
- 等等

LLM 看到流年 entry 的合化 tag，从 entry 在流年位置 + 大运 ganzhi 渲染在大运行的上下文，自然推断 cross source。**不需要"跨大运"语义标签** —— context 已经足够。

## 工程方法学（贯穿整个系列）

3 天交付 7 个 plan + 1 个 audit cancellation 不是偶然。每个 plan 都跑同一个流水线：

1. **Brainstorming skill** — 一次 1 个问题，多选 + 推荐选项 + 1 句理由，scope 决策必须 user 确认
2. **Spec 文档** — 12 节 + 2 个 Appendix（小 plan 5-6 节）
3. **Writing-plans skill** — TDD-style task 分解，每 task 含 bite-sized step + 完整代码 + 确切命令 + commit message 模板
4. **Codex 外部执行** — verification-first（task 4 必须先验证 input 真触发再写测试）
5. **Review** —  每轮 sanity math 预先 hand-verify, codex 输出 byte-perfect 比对
6. **Inline plan 修正** — codex catch 我的 plan typo (sizhu vs bazi / day_score vs same_ratio / test path / classifier off-by-one) 都 inline 改 plan + 单独 commit
7. **Push** — 每 task 独立 commit + push, golden case verify-first 防 silent assertion loosening

3 天里 codex catch + plan 修正大概 8-10 次，没一次 silent passing。这种"诚实 catch"是质量的真正保护。
