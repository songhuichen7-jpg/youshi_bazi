## 用户感知变化

### 以前（Plan 7.5b 及之前）

**问题 1 — 极端盘的扶抑用神过于粗糙**：
任何日主弱到"极弱"级别（sameRatio ≤ 0.12）的盘，Plan 7.3 li_liang 只 emit `身弱`，fuyi_yongshen 返回 `"印 / 比劫"`（两选一）。LLM 看到就只能说"补印或补比劫"，没法写"这么弱需要双补"。

**问题 2 — 多元素用神 max 偏乐观**：
命局用神是 `"甲木/戊土/庚金"` 三元素的盘，行运 庚申 同时：
- 克甲木 -4（木被毁）
- 泄戊土 -2（土被消）
- 比助庚金 +2（金得气）

Plan 7.4 `max(sub_scores) = +2 → '喜'`。LLM 看到"这步运喜"，但实际上 木+土 两元素都被猛克。用户体感"为啥引擎说喜我却觉得不对劲"。

**问题 3 — 合化任意位置**：
命局 4 干（年、月、日、时）任意两干合化都触发。古籍 strict reading "干合须相邻"。Plan 7.4 任意位置触发率偏高，虽然短期对评分影响小，但为 Plan 7.7 cross interaction 预埋问题。

### 现在（Plan 7.6 上线后）

**#1 修 极端盘**：li_liang 5-bin 激活，sameRatio ≤ 0.12 的盘 → `极弱` → Plan 7.3 FUYI_CASES 的 `"印 + 比劫（同扶）"` compound rule fire。LLM 现在可以写：

> 辛金生在卯月，木旺金囚，日主本身偏弱。按《滴天髓》的思路，"衰者喜帮喜助"，土（印）和金（比劫）能扶住你的本体。

相比原来"补印或补比劫"（两选一）升级为"双扶"（同时补），叙事精度跃升。

**#2 修多元素**：`max → weighted average` (0.5/0.3/0.2 归一化)。前文案例 庚申 vs 甲木/戊土/庚金：
- 旧：max(-4, -2, +2) = +2 → 喜
- 新：0.5·-4 + 0.3·-2 + 0.2·+2 = -2.2 → round -2 → **忌**

新评分**如实反映三元素合成作用**。LLM 输出里自然说"三路用神指向不同方向，得看你问的是什么事"。

**#3 修 adjacency**：命局内部合化要相邻（年-月、月-日、日-时 3 对）；大运/流年 vs 命局保持任意位置（外来作用力不论位置）。实际审核发现 Plan 7.4 没有命局内部合化的 call site —— adjacency 改动是**forward-compat 空改动**，留给 Plan 7.7 cross interaction 使用。

## 核心机制

### #1 合化 adjacency 严格化

```python
# xingyun.py
def _detect_ganhe(
    gan: str,
    mingju_gans: list[str],
    *,
    source_idx: int | None = None,   # NEW
) -> str | None:
    for idx, mg in enumerate(mingju_gans):
        if mg == gan: continue
        if source_idx is not None and abs(idx - source_idx) != 1:
            continue   # 命局内部严格相邻
        ...
```

`_detect_liuhe` 同模式。Plan 7.4 所有 callers 都是外来 (source_idx=None)，行为不变。

### #2 多元素用神 weighted average

```python
# xingyun_data.py
YONGSHEN_WEIGHTS: list[float] = [0.5, 0.3, 0.2]

# xingyun.py score_yun multi-element reduction:
# Plan 7.4: final_score = max(sub_scores)
# Plan 7.6: final_score = round(Σ weight_i × sub_score_i) 归一化
```

单元素：权重 [1.0]（行为不变）  
2 元素：[0.625, 0.375]  
3 元素：[0.5, 0.3, 0.2]  
4+ 元素：第 4 权重 0（截断保护）

`winningYongshenElement` 仍取 max sub_score 元素（for explainability）。

### #4 li_liang 5-bin

```python
# li_liang.py — 5 named THRESHOLD_* constants (refactor + extend)
THRESHOLD_JI_QIANG = 0.76     # NEW (Task 0 p95)
THRESHOLD_SHEN_QIANG = 0.55   # existing inline → named
THRESHOLD_ZHONG_HE = 0.35     # existing inline → named
THRESHOLD_JI_RUO = 0.12       # NEW (Task 0 p5)
THRESHOLD_CONG_CANDIDATE = 0.15  # existing inline → named

# [0, 0.12) 极弱 | [0.12, 0.35) 身弱 | [0.35, 0.55) 中和 |
# [0.55, 0.76) 身强 | [0.76, 1] 极强
```

阈值来自 Task 0 pre-task sampling（N=1000 random birth_inputs，seed=42，city=北京）的 p5/p95。分布 unimodal，range [0.00, 0.91]，median=0.42。

5-bin 激活后 Plan 7.3 FUYI_CASES 的 `极弱 → '印 + 比劫（同扶）'` 和 `极强 → '官杀 + 食伤（双泄）'` 规则 fire（原本 dead code）。

## 关键 audit 发现 / catch

### Codex Task 0 catch — spec/plan 字段名错误

我 spec 写 `day_score` (signed int)，plan 用它搭 classifier。Codex Task 0 实际读 li_liang.py 发现：
- 不存在 `day_score` 字段
- 现有 dayStrength 分类用 `same_ratio` (比值 [0, 1])
- 现有阈值是**inline literals** (0.55 / 0.35 / 0.15) 无 named constants

Spec + plan inline 修正 (`61833a0`)：day_score → same_ratio，添加"FIELD NAME CORRECTION"备注。没这个 catch，Task 1 实施会撞墙（给了 Task 0 strong justification 作为 pre-task ≠ 过度设计）。

### Codex Task 1 catch — test 路径

我 plan 写 `paipan/tests/test_force.py`，实际路径 `paipan/tests/unit/test_force.py`（有 `unit/` 子目录）。Plan 修正 (`7f1a60b`)。

### Downstream fallout 比 spec 估计少一半

Spec §6.1 估计 Plan 7.4/7.3/7.5a/7.5b golden 会有 ~5-15 case shift。实际只 2 个 case：
- `test_analyzer_parity[H_sanhui]`：dayStrength `身弱 → 极弱` (hand-verify: sameRatio=0.09 < 0.12)
- `test_multi_element_yongshen_takes_max_score`：max → weighted avg 的直接反转 (hand-verify: -2.2 round → -2 忌)

原因：Plan 7.4 10 个 GOLDEN_XINGYUN_CASES 都是单元素 primary（weighted avg 对它们行为不变）；Plan 7.3 系列 golden 大多中性 dayStrength 盘，没触到 p5/p95 边界。

## 限制 + Reserved 后续

### Plan 7.7 (reserved)
**大运-流年 cross interaction 评分** —— 当前 Plan 7.4 评分只对"大运干支 vs 命局"和"流年干支 vs 命局"分别算。古籍还有"大运干支 vs 流年干支"直接作用（大运庚 + 流年乙 → 乙庚合化金）。这层 Plan 7.6 没做，LLM 需要自己从 mechanisms 里推。Plan 7.7 加这层后会更准。

体量 ≈ Plan 7.4（新评分维度 + 新 mechanism tags + 新 golden cases），需要独立立项。

### Plan 7.6 已知未做
1. **adjacency forward-compat 未触发** — 所有 Plan 7.4 callers 都是 external，内部命局合化目前零 call site。Plan 7.7 cross interaction 可能是第一个用到的。不是 bug，是 infra 先行。
2. **weighted avg 没有 configurability** — 权重硬编码 [0.5, 0.3, 0.2]。如果以后某用户坚持 max 读法，需要加 setting。v1 不做。
