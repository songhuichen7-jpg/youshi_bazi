## 一句话总结

清理 Plan 7.4 + 7.5a 留下的 4 个 quick-win polish 项 —— 1 个语义对齐 fix + 1 个 utility 加 + 1 个 14-处重构 + 1 个 SMS test flake 根因修复。无算法行为变化，所有 7.3 / 7.4 / 7.5a 行为契约不破。

## 4 个 ship 项

### #1 `transmuted.from` 用 GEJU_RULES key

**之前**：`transmuted.from = '劫财（自立）'`（candidate name 来自 GEJU_RULES 规则的 name field）
**现在**：`transmuted.from = '劫财格'`（post-alias GEJU_RULES key，跟 `transmuted.to` 同命名空间）

LLM 渲染从 `劫财（自立） → 比肩格`（混合 candidate + 格局名）变成 `劫财格 → 比肩格`（两边都是格局名）。1-line fix。

### #4 `_trim_note` 中文标点边界截断

替换 `score_yun()` 的 naive `note[:30]` 为 `_trim_note()`：超 30 字时优先在中文标点边界截断（`。 > ；： > ，`），fallback 到字符切（如果标点在前半截就放弃避免切得太短）。Plan 7.4 ship 时的实际 note 都没超 30，所以这层 mostly serves future-proofing，但底层算法 robust 了。

### #3 `mechanism_tags.py` 中心化

新 module 14 个 symbol（10 constants + 4 builder functions）。`xingyun.py` 12 处 f-string 替换为 import 常量/builder。Byte-for-byte 输出不变（byte-identity test 验证）。`GAN_HAO`/`ZHI_HAO` 2 个常量声明备用（Plan 7.4 实现里"相耗" case 没真触发，但 vocabulary 完整）。

未来要改 mechanism 文案、查 mechanism 类型分布、或加 mechanism-based 渲染逻辑（"这一段以合化为主"），都只动这一个文件。

### #2 SMS test flake 根因修复

**Symptom**：`test_send_sms_code_does_not_charge_when_user_none` 在全 suite 跑偶尔红、隔离跑绿。Plan 7.4 + 7.5a Task review 都看到过。

**Codex debug** 找到 deterministic ordering：`test_quota_route::test_quota_reflects_sms_usage` 跑在前面就 100% trigger。

**Root cause**：integration test commit 真实 `quota_usage(kind='sms_send')` row 到共享 DB；unit test 错误地 assert 全局 `count(*) == 0`（assert "DB 干净" 而不是 "这个 call 没 insert"）。

**Fix**：localized 到 flaky test 自己的 per-test transaction 里清掉 preexisting rows。**production code 零改动**。没走"全局 fixture"或"refactor service"等更大 scope 选项 —— 因为根本不是隔离问题，是 assertion 语义错（global vs delta）。

**Verification**：5 轮 random-order full-suite runs 全绿（130s / 168s / 72s / 68s / 64s 实际跑时）。

## 关键 audit 发现 / catch

### Codex Task 2 fixture catch
我 plan 里 `test_trim_note_long_with_comma_cuts_at_comma` 的 fixture 有 commas 在 idx 4 和 10，**两个都在 limit//2=15 之前**。`_trim_note` 算法故意 reject 前半截标点（避免切太短），所以这个 fixture 实际跑 fallback to char cut，**没测到 comma branch**！codex 自己构造了 idx=16 的真后半截 fixture（"丙生用神调候扶抑兼顾格局仍偏燥些，..."）才正确命中。Plan 已 inline 修正 (b16b532)。

### Codex Task 3 grep catch
我 plan 说会有 14 处 f-string 替换。实际 grep 出 12 处 active —— `相耗` (GAN_HAO/ZHI_HAO) case 在 Plan 7.4 实现里没真触发，所以没 f-string。Codex 按 spec 完整声明了 2 个 unused 常量（vocabulary completeness），但只替换了 12 个 active site。Spec 的 "Subtle gotchas" 已经预警过此情况。

### Codex Task 4 debug discipline
完整走完了我 plan 的 5 步 debug workflow（reproduce → root cause → minimum-scope fix → verify → regression）。最关键的是它**没误诊为隔离问题**走 option (b)/(c)，而是识别到这是 assertion 语义 bug。这种判断力是 polish round 该有的样子。
