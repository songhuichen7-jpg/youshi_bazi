补完 Plan 7.5a 的 time-axis counterpart：当大运/流年支与命局支 + 月令凑成完整 三合 / 三会，那段时间的 effective 格局质变，per-entry `transmuted` 字段挂在 `xingyun.dayun[i]` / `liunian[k][j]`。LLM 现在能写出"大运 X 期间格局变了"的时间轴叙事。Plan 7.4 行运评分契约保持，命局用神 anchor 不变。

## 用户感知变化

### 以前（Plan 7.4 + 7.5a）
用户问 "我大运里某段格局会变吗"：
- Plan 7.5a 只检测**命局自带**的合局触发（少数命局适用）
- 大多数盘的命局自带不触发 transmutation → LLM 答 "你格局是 X 格，未来不会变"
- 但实际上很多盘是 "命局少 1 支，大运/流年来补全才触发" → 这种动态变化 LLM 完全看不到

### 现在（Plan 7.5b 上线后）
```
在你走过的所有大运里，只有 9-18岁乙酉运 那段明确变了格局。为什么？
《子平真诠》讲「八字用神，专求月令」，你原局月令是申金正财，但大运地支
酉和原局申、加上戌中辛金，三会西方金局，金气旺到极致，庚金偏财透出力量
压过正财，所以格局从正财格转成了偏财格。

那十年金旺生水，食伤生财的路径被强化，对应到现实里，可能早年就显露出
经商或灵活求财的苗头。

其他大运...都没有这种动摇月令的会局或冲克。格局还是正财格或食伤生财格
的框架，变化在于大运干支对用神甲木、丙火的补益或损耗——比如己丑运湿土
晦火，用神丙火被泄，这十年底色就偏平缓...
```

LLM 看着引擎注入的 ⟳ 月令变化 块（`正财格 → 偏财格  三会西方`），按 ZPZQ 风格写出"质变"叙事，区分 **"格局变化"** vs **"格局发挥状态变化"** 这种 second-order distinction —— 这是 Plan 7.5a 静态 transmutation + Plan 7.5b 动态 transmutation + Plan 7.4 行运评分**三层信号的真正合奏**。

## 核心机制

### 检测条件
对每条大运/流年 entry，检查是否：
1. 大运/流年支 + 命局支 + 月令 凑齐完整 三合 / 三会
2. 月令必须参与（ZPZQ ch10 strict）
3. 该合局尚未被 baseline (Plan 7.5a 静态 / 大运 transmuted) 处理

### Per-entry shape
`xingyun.dayun[i].transmuted` 和 `xingyun.liunian[k][j].transmuted` 是 optional dict，跟 Plan 7.5a `yongshenDetail.transmuted` 字段 byte-for-byte 一致：
- `trigger`: type/wuxing/main/zhi_list/source
- `from`: 原本格局名 (post-alias GEJU_RULES key)
- `to`: 虚拟格局名
- `candidate`: 重算的格局法 用神
- `warning` + `alternateTriggers`

### Dedup 关键设计
两层 dedup 防止 redundant 触发：

**大运层 dedup against 命局-only baseline** (避免重复 Plan 7.5a 静态)：
- 命局 [酉,亥,卯,未] 自带亥卯未三合 (Plan 7.5a fire 静态)
- 大运任意支加进来 → with-大运还是同一个亥卯未 → **dedup return None**
- 让 7.5a 在 yongshenDetail 处理，7.5b xingyun 不重复

**流年层 dedup against 大运 transmuted** (避免大运 echo)：
- 大运 transmuted 已 fire 三合A
- 流年支恰好是合局支之一 → with-流年仍是同一三合A → **dedup return None**
- 只有流年触发**不同**combo时才 fire

`_is_same_combo` 用 (trigger.type + set(zhi_list)) 结构化比较，对未来 source string 重命名 robust。

## 接口

```python
chart.paipan.xingyun.dayun[i] = {
    'index': i, 'ganzhi': '...', 'startAge': ..., 'startYear': ..., 'endYear': ...,
    'label': '喜', 'score': 2, 'note': '...', 'mechanisms': [...],
    'isCurrent': bool,
    'transmuted': None | {                          # NEW
        'trigger': {...}, 'from': '...', 'to': '...',
        'candidate': {...}, 'warning': str | None,
        'alternateTriggers': [],
    },
}

chart.paipan.xingyun.liunian['3'][2] = {
    'year': 2026, 'ganzhi': '...', 'age': 34,
    'label': '忌', 'score': -2, 'note': '...', 'mechanisms': [...],
    'transmuted': None | {...},                     # NEW (dedup'd)
}
```

`build_xingyun()` 加 optional `chart_context` kwarg；不传时 `transmuted=None`，行为退回 Plan 7.4。

## 关键 audit 发现 / catch

### Codex Task 4 发现 1993 chart 不能做 all-None oracle

我 spec/plan 写"标准 1993-07-15 chart 应该所有 transmuted 都 None" —— 但 codex verification-first 跑出来发现这是错的。1993 chart 命局 [酉,未,酉,未] + 月令未，**流年带巳/午 → 巳午未三会火** 或 **流年带亥/卯 → 亥卯未三合木** 都会真触发 dynamic transmutation。

Codex **没硬塞测试让通过**，而是：
1. 验证报告找出 4 个真触发的流年（辛巳/甲午/辛亥/辛卯）
2. 替换 fixture 为真正 all-None 的 1989-08-15 北京 chart
3. 全程 disclosure

我顺手修了 spec + plan 文档反映这一发现 (`e4f9393`)。这是 verification-first workflow 防止假阳性 oracle 的真实价值。

### Plan 7.5a 引擎核心 100% 复用

`_detect_transmutation` 和 `_compute_virtual_geju_name` 完全没改，只在 xingyun.py 加了 `_detect_xingyun_transmutation` wrapper 调它们。这是 Plan 7.5a 设计时就 anticipate 的 reusability —— spec §3.1 一开始就说"架构上 100% 复用"，Task 1 起就这样实施。

### Task 4 verified diversity

5 个 verified golden case 都从 1970-1971 systematic search (year/month grid) 找到，覆盖：
- **trigger types**: sanHe + sanHui 都有
- **trigger wuxings**: 木 / 火 / 金 / 水 (4 种)
- **dayun + liunian 双层触发**: 每个 case 两层都 fire (5/5)

## 限制 + Reserved 后续

### Plan 7.5b 已知限制
1. **流年 dedup 用 `_is_same_combo` 结构化比较** — robust against source 重命名 ✓
2. **命局支跟大运/流年支重复** (e.g. 大运卯 + 流年卯 + 命局含卯) — 当前 detection algorithm 用 `if z in zhis` set-membership，重复支不影响。但如果未来 path 用 list count 可能需 dedup
