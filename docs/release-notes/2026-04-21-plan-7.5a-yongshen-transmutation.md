## 用户感知变化

### 以前（Plan 7.3 + 7.4）

用户问 "我命局格局是什么？"（如果他/她的命局自带 三合木局）：
```
你是乙木日主，月令寅木...本气甲木为劫财，所以是月刃格...
```
LLM 看着 Plan 7.3 输出的"月刃格 / 劫财（自立）"回答，没意识到地支寅卯辰会成的木局已经把月令"质变"为完整的木方局，整个格局应该按建禄看。

### 现在（Plan 7.5a 上线后）

```
你这盘命局的格局确实发生过一次根本性的变化，从结构上看，是月令的"身份"被
地支的会局改写了。

月令寅木，本气是甲木劫财...你年支申、月支寅、日支卯、时支辰，暗中构成了
"寅卯辰"三会东方木局。

《子平真诠》论方局时点明："支会方局，则月令之气随之而变。"...

因此，你的格局核心从原本的"劫财（自立）"，质变成了"比肩（自立）"，也就是
常说的建禄格。古籍对此描述为"建禄无辅，专凭自身担当"...

这个变化是根本性的。因为格局的喜忌取用完全翻转了：正财格要"顺用"...建禄格
要"逆用"或"自立"...
```

LLM 看着引擎注入的 transmuted 块（`劫财（自立）→ 比肩格 / 三会东方 / 比肩
（自立）/ 子平真诠·论建禄月劫`）按 ZPZQ 风格写出"由 X 变 Y"的叙事 + 解释为什么这个变化"根本性"。

## 核心机制

### 检测条件
命局支序列里 **月令 + 命局其余两支** 凑齐一组完整 三合（4 种）或 三会（4 种）—— 月令必须参与，否则不算月令质变（ZPZQ ch10 strict reading）。

### 处理范围
- **重算**：仅格局法
- **不重算**：调候法（keyed on 真实地支位置） + 扶抑法（keyed on dayStrength）
- **primary 不变**：仍按 Plan 7.3 三法投票，保持 backward compat

### 虚拟格局命名
五行 + 日主 + main 支 → 10 个格局名之一（5 十神类 × 2 polarity）。例：丁火（阴）+ 亥卯未三合木局（main=卯阴乙）→ 木生火 = 印 + 阴阴 = same → **偏印格**。

## 接口

```python
yongshenDetail.transmuted = {              # optional, only when triggered
    'trigger': {
        'type':     'sanHe' | 'sanHui',
        'wuxing':   '木',                   # 合局五行
        'main':     '卯',                   # main支 (三合) or 主气支 (三会)
        'zhi_list': ['亥', '卯', '未'],
        'source':   '三合亥卯未局',          # 人可读名
    },
    'from':              '原本格局',         # Plan 7.3 算出的格局候选名
    'to':                '虚拟格局名',       # 重算后的格局名 (10 之一)
    'candidate':         {...},              # 新格局对应用神 (geju_yongshen 输出)
    'warning':           '...' | None,       # 与原取用法分歧时的警告
    'alternateTriggers': [],                 # 罕见多触发时其他备选 (v1 不渲染)
}
```

`primaryReason` 字段也加 hint：`"...（注：月令合局触发格局质变，详见 transmuted 字段）"`。

## Audit 关键发现

### 1. 测试用 birth_input estimate vs 真盘验证 (Task 4)

我 spec §6.2 列了 5 个候选 birth_input 作为 golden case。Codex 强制走 verification-first workflow（Step 4.1），结果发现 **5 个里 4 个根本不触发**：

| 原 candidate | 替换为 |
|---|---|
| 1983-11-04 0:00 male 北京 (丁火亥月) | 1980-11-14 14:00 male 北京 (辛金亥月) |
| 2014-06-10 14:00 male 广州 (戊土午月) | 1980-06-10 20:00 male 北京 (甲木午月) |
| 2010-02-20 12:00 female 深圳 (丙火寅月) | 1980-02-12 08:00 male 北京 (乙木寅月) |
| 1993-07-15 14:30 male 长沙 (negative control) | 1980-05-09 14:00 male 北京 (壬水巳月) |

最终 5 个 verified case：3 sanHe + 2 sanHui，覆盖 木/水/火 三个 wuxing。Codex 的 disclosure 全程透明，没硬塞 dead inputs。

**这是 verified-first workflow 的价值证明**：spec 写作时的"应该 work 的 case"和真实数据 distribution 经常不对，必须实测。

### 2. Unknown-hour guard 修正 (Task 1)

Codex 读 `analyzer.py:30` 发现 `_split_pillar(None)` 返回 `{"gan": None, "zhi": None}` —— 所以 `h` **永远是 dict，从来不是 None**。我 plan 写的 guard `if h else []` 永远 truthy，会把 `None` 静默注入 `mingju_zhis`，下游 wuxing lookup 会 KeyError。Codex 改成 `if h["zhi"]` 直接判 zhi 非空。

### 3. transmuted.from vs transmuted.to 语义不对称 (queued for 7.5a.1)

`from` 取的是 `composed.candidates[格局].name`（candidate 名，如 "劫财（自立）"），`to` 取的是 `virtual_geju_name`（格局名，如 "比肩格"）。两者概念层级不一样。Spec §4 原意是 `from` 也应该是格局名。

LLM 没被这个 confusing —— smoke 里它正确推断"原劫财格 → 现比肩格"。但视觉上 `劫财（自立） → 比肩格` 不工整。

**严重度 low** —— 不影响功能，留 Plan 7.5a.1 polish。
